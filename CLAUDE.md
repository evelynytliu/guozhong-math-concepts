# 國中數學「概念理解」互動教材網站 — CLAUDE.md

## 專案目的

為一位即將升國一的孩子建立**數學概念理解教材網站**。

**這個孩子的核心問題（這份規格的設計起點）：**
> 習慣「背公式、看題型反射解題」。在熟悉題型上表現好，但題目稍微變化（換問法、換情境、多一個轉折）就會做錯。根本原因是「解題時腦中跑的是檢索記憶，而不是推理概念」。

**因此本網站的目標不是「給更多題目」，而是：**
1. 讓孩子**看到概念是怎麼推導出來的**（而不是直接給公式）
2. 強迫他**用自己的話解釋為什麼**（背公式的孩子最怕被問「為什麼」）
3. 用**變形題**驗證他是真懂還是在背（同一概念、不同外觀）

> ⚠️ 這個網站跟既有的「學習打卡 + 錯題本 App」是**兩個獨立專案**。錯題本是「事後診斷錯在哪」，這個教材網站是「事前把概念打扎實」。兩者互補，不要合併。

---

## 使用者角色

- **學生（國一孩子）**：閱讀概念教材、做引導推導、寫解釋、做變形題驗證
- **家長（媽媽）**：不需要每天介入，偶爾看孩子在哪個單元卡住

這個專案**單純自用**，不需要多人帳號系統、不需要部署給外人。

---

## 技術棧

延續既有專案習慣：
- **Next.js 14+** App Router
- **Tailwind CSS + shadcn/ui**
- **Supabase**（儲存學習進度、孩子寫的解釋文字；免費方案足夠）
- **本地優先（local-first）執行**：因為有 AI 環節要吃訂閱額度（見下方「AI 環節」），後端跑在自己電腦、用 `claude login` 的訂閱認證，**不部署到 Vercel**（部署雲端就無法使用訂閱認證）。用的時候 `npm run dev` 啟動即可。

## 🎯 核心教學結構（整個網站的靈魂）

**每一個概念單元，都必須照這個固定的五段式流程設計。** 這個流程就是用來破解「背公式」的，不可省略、不可改順序。

### 第 1 段：情境引入（為什麼需要這個概念）
- 用一個生活化的問題開場，讓孩子感覺到「現有的工具不夠用，需要新東西」
- 例：在學「負數」前，先問「溫度比 0 度還低怎麼記？欠錢怎麼記？」
- **目的**：讓概念是「被需要而生」的，不是憑空背的規則

### 第 2 段：引導推導（自己把規則「長」出來）
- **不直接給公式或規則。** 用一連串小問題，引導孩子一步步自己推出規則
- 每一步都是孩子可以互動作答的（填空、選擇、拉動數線等）
- 例：學「負負得正」，不是給規則，而是用數線/欠債情境讓他自己發現「拿走負的等於加上正的」
- **目的**：讓他經歷「規則是怎麼來的」，而不是接收一條空降的公式

### 第 3 段：用自己的話解釋（這段是破解背公式的關鍵）
- 推導完後，要求孩子**打字寫下**：「用你自己的話說，為什麼這個規則會成立？」
- 兩種檢查模式（見下方「檢查機制」）：
  - **靜態模式**（大多數單元）：他寫完後，網站展開「參考解釋」讓他自己對照，並自評「我的解釋有沒有講到重點」
  - **AI 模式**（關鍵單元）：他寫完後送給 AI，AI 判斷他是「真的理解」還是「在複述背起來的句子」，並回饋追問
- **目的**：講不出為什麼 = 還是在背。這段直接逼出來。

### 第 4 段：變形題驗證（同概念、不同外觀）
- 給 3 題：**第 1 題跟教材長得像，第 2、3 題換情境/換問法/多一個轉折**，但考的是同一個概念
- 每題做完，不只給對錯，要顯示「這題其實在考剛剛那個概念的哪個面向」
- 如果第 1 題對、第 2/3 題錯 → 系統明確指出「你會這個題型，但概念還沒遷移」，這正是這個孩子的典型狀況
- **目的**：破解「看題型解題」。變形題就是「同概念換外觀」，背題型的人會在這裡現形。

### 第 5 段：回扣（口頭解釋提示）
- 單元結尾顯示一句提示給家長/孩子：
  > 「現在闔上網站，用嘴巴跟家人解釋一次這個概念。講得出來才算真懂。」
- **目的**：口頭表達是理解的最終驗收，背公式的孩子最怕這關。

## 檢查機制：靜態 vs AI（混合策略）

> 這是經過刻意設計的成本控制：常見單元用零成本的靜態引導，只有最容易卡概念的關鍵單元才動用 AI。

### 靜態模式（預設，大多數單元用這個）
- 第 3 段「用自己的話解釋」：孩子寫完 → 點按鈕展開**預先寫好的參考解釋** → 孩子自評（三選一：我有講到 / 部分有 / 我其實講不太出來）
- 自評為「講不太出來」→ 網站建議他回到第 2 段重做一次推導
- **完全不呼叫 AI，零成本，離線也能用**

### AI 模式（只在標記為「關鍵單元」的地方啟用）
- 第 3 段：孩子打字解釋 → 送 Claude API → Claude 回傳：
  1. 判斷：他是「理解型解釋」還是「複述型解釋（在背）」
  2. 一個追問：針對他解釋裡最薄弱的點再戳一下（例如「那如果數字換成負的，你的說法還成立嗎？」）
  3. 鼓勵語：給孩子看的、正向的一句
- **技術實作**：在 Next.js 的 API route 裡用 Agent SDK（TypeScript），把工具關掉，純粹當分析引擎。用本地 `claude login` 訂閱認證（所以不部署雲端）。
- **AI 回傳請用 JSON 結構**，方便前端渲染：
  ```json
  {
    "understanding_level": "理解型 | 複述型 | 不確定",
    "followup_question": "針對弱點的一個追問",
    "encouragement": "給孩子的正向回饋"
  }
  ```
- API route 要 try/catch，AI 失敗時自動退回靜態模式（顯示參考解釋），不能讓孩子卡住。

## 首批內容（國一上學期，3 個單元）

挑選原則：**最容易被「背公式」帶過、且國中後續會一直用到的概念。**

### 單元一：負數與數線 〔靜態模式〕
- **核心概念**：負數是什麼、數線上的位置、絕對值的意義
- **最容易被背的點**：「負負得正」「絕對值就是去掉負號」——孩子常背規則但不懂意義
- **引導推導重點**：用數線移動、欠債/溫度情境，讓他自己推出加減規則，而不是背「同號相加、異號相減」

### 單元二：一元一次方程式的「應用題」 〔AI 模式 — 這是最關鍵單元〕
- **核心概念**：怎麼把一段文字「翻譯」成方程式（設未知數、找等量關係）
- **最容易被背的點**：這正是「題目一活就錯」的重災區。背題型的孩子會記「年齡問題這樣設、雞兔同籠那樣設」，一旦情境變了就不會
- **引導推導重點**：教「找等量關係」這個通用思維，而不是分類記題型。第 4 段變形題刻意用「同一個等量關係、包裝成完全不同情境」來驗證
- **為什麼這個單元接 AI**：應用題的「設未知數理由」最需要孩子用自己的話講清楚，靜態對照很難判斷他是真懂，值得花 AI 額度

### 單元三：因數與倍數 / 最大公因數與最小公倍數 〔靜態模式〕
- **核心概念**：為什麼要分解質因數、GCD 和 LCM 的實際意義
- **最容易被背的點**：背「短除法」的操作步驟，但不知道為什麼這樣做、什麼時候該用 GCD 什麼時候用 LCM
- **引導推導重點**：用「磁磚鋪滿地板（GCD）」「兩班公車同時發車（LCM）」這種情境，讓他理解這兩個東西在解決什麼問題

> 之後可依孩子進度，用同一套五段式結構繼續加單元（整數四則運算、座標平面、比與比例式…）。內容設計時**永遠先問「這個單元最容易被背的點是什麼」**，再針對那個點設計引導推導與變形題。

## 暑假作業引導導師（多科擴充）

> 這個網站除了上面的「數學概念五段式單元」，還擴充成**各科暑假作業的引導導師**。
> 作業來源：`參考文件/竹光2026暑假作業.pdf`（2026 竹光國中新生暑假作業）。
> 內容放在 `src/content/homework/`，路由 `/homework/[id]`，與數學單元並存於首頁。

**最高原則（不可省略）：這裡只是「打草稿、想清楚」的地方。草稿完成後，一定要提醒孩子親手抄到作業本上，手寫的才算交作業。** 草稿區全程都有這個提醒，組稿頁有明顯的手抄提示與「我已抄到作業本上」確認。

作業分兩種型態（`HomeworkKind`）：

- **草稿 builder（`draft`）**：國文讀書報告、數學生活概念、生物五大類、社會成長小書。
  每一題給一組「**不同角度的提示（sparks）**」——點一下幫他開頭、填空，他再用自己的話接下去，
  草稿慢慢成形。**不給標準答案**（呼應這份規格的起點：破解「照範例反射、照抄」）。
  每個欄位另有「AI 追問」按鈕（混合模式）：送 `/api/coach`，依他寫的內容給幾個方向 + 一個追問，
  **絕不直接幫他寫成品**；AI 連不上一律自動退回靜態 sparks（跟 `/api/explain` 同一套 try/catch fallback）。
- **單字 drill（`vocab`）**：英文單字表。看中文、拼英文，**答錯的字自動回鍋，整組每個字都拼對才過關**
  （這就是使用者要的「確保每個單字都答對」）。批改容錯大小寫/空白/標點，只看字母；
  同一中文對多個英文時拼出任一個都算對。單字清單在 `vocab-list.ts`（已對照 PDF 校正拼字）。

> 加新作業：在 `src/content/homework/` 仿照現有檔案寫好，加進 `index.ts` 的 `homeworks` 陣列即可。
> 設計草稿題時永遠記得：**提示是「不同角度」不是答案**，且最後一定要回到「手抄到作業本」。

## UI / 設計方向

- **給 12 歲孩子用，但不要幼稚。** 乾淨、清楚、有呼吸感，不要花俏到分心。
- 五段式流程用**明顯的進度指示**（第 1 段 → 第 5 段），讓孩子知道自己在哪、還剩多少。
- 第 2 段「引導推導」是互動重點，盡量做成可點、可拉、可填空，不要只是讀文字。數線可以用簡單的 SVG 或 div 做互動。
- 第 4 段變形題答錯時，**回饋語氣要鼓勵**，重點放在「這代表概念還沒遷移，不是你笨」。
- 配色建議溫和不刺眼（淺色底、單一主色點綴），避免大紅大綠的對錯燈號造成壓力。
- 參考 `frontend-design` 的設計原則，避免做出一看就是模板的樣子。

## 開發順序（建議給自己的里程碑）

1. **建專案 + CLAUDE.md**：Next.js repo，寫清楚「本地跑、AI 環節用 Agent SDK、不部署雲端」
2. **搭五段式流程的殼**：先用「單元一（負數，靜態模式）」把五段式 UI 流程完整跑通，不接 AI
3. **接 Supabase**：存進度、存孩子寫的解釋
4. **做單元三**（因倍數，也是靜態）：驗證「同一套結構能套用到不同單元」
5. **接 AI（單元二）**：API route + Agent SDK，做應用題單元的 AI 解釋判斷，含失敗退回靜態的 fallback
6. **打磨**：進度指示、變形題回饋語氣、家長偶爾查看的簡單入口

## 給 Claude Code 的重要提醒

- **每個單元都必須照五段式結構做，這是專案的核心價值，不是裝飾。** 如果某個設計偷懶變回「講解 + 練習題」的傳統教材，就失去意義了。
- **第 3 段（用自己的話解釋）和第 4 段（變形題）是這個網站存在的理由**，這兩段做扎實，其他都是配角。
- AI 環節**只**在 `check_mode === 'ai'` 的單元啟用，不要每個單元都接 AI（會浪費訂閱額度）。
- AI 呼叫一律要 try/catch + 退回靜態模式，不能讓孩子因為 AI 出錯就卡住。
- 內容（推導步驟、變形題、參考解釋）可以先放在程式碼裡的結構化資料（例如每個單元一個 `.ts` 設定檔），方便日後手動增修，不用一開始就做後台編輯介面。
